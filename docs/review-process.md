# 정식 검수 프로세스 설계 (L1→L2→L3)

> Version: 1.0.0
> 작성일: 2026-03-12

---

## 1. 검수 흐름

```
데이터 입력/수집
      │
      ▼
┌─────────────┐     실패     ┌──────────────┐
│ L1 데이터    │ ──────────→ │ QA 수동 검토   │
│ (자동+QA)    │             │              │
└──────┬──────┘             └──────┬───────┘
       │ 통과                      │ 수정 완료
       ▼                          ▼
┌─────────────┐     반려     ┌──────────────┐
│ L2 과학 검수  │ ──────────→ │ 수정 요청      │
│ (의학/약학)   │             │ → L1부터 재시작 │
└──────┬──────┘             └──────────────┘
       │ 승인
       ▼
┌─────────────┐     반려     ┌──────────────┐
│ L3 규제 검수  │ ──────────→ │ 수정 요청      │
│ (RA 자문)    │             │ → L2부터 재시작 │
└──────┬──────┘             └──────────────┘
       │ 승인
       ▼
  is_published = TRUE
  (서비스 노출)
```

## 2. 검수 레벨 상세

### L1: 데이터 검수 (자동 + QA)

**담당**: 시스템 자동 + QA 담당자
**SLA**: 즉시~1일
**목적**: 데이터 형식·완전성·정합성 확인

#### 자동 검증 규칙

| 규칙 | 대상 | 동작 |
|------|------|------|
| 필수 필드 누락 | 모든 엔티티 | 실패 → QA 수동 |
| 단위 유효성 | dosage, product_ingredients | mg/mcg/IU/g 외 → 실패 |
| 숫자 범위 이상 | dosage, 함량 | 비정상적 고값 → 실패 |
| PMID 중복 | evidence_studies | 같은 PMID 존재 → 실패 |
| 금지 표현 탐지 | claims, 라벨 텍스트 | 금지어 사전 매칭 → 실패 |
| checksum 변경 없음 | raw_documents | 이전과 동일 → 스킵 (변경 없음) |
| confidence_score | extraction_results | < 0.70 → 실패 |
| 원료명 매핑 실패 | product_ingredients | canonical 매칭 없음 → 실패 |

#### 결과 처리
- `auto_check_passed = TRUE` → L2 review_task 자동 생성
- `auto_check_passed = FALSE` → `auto_check_details`에 실패 규칙 기록, QA 할당
- QA가 수동 수정 후 재실행 가능

---

### L2: 과학 검수 (의학/약학 감수)

**담당**: 의학/약학 전문가
**SLA**: 3~5일
**목적**: 과학적 정확성·해석 타당성 확인

#### 검토 항목

| 항목 | 검토 내용 |
|------|-----------|
| **논문 해석** | 결론이 원문과 일치하는지, 과대 해석 여부 |
| **근거 등급** | A~I 등급이 연구 설계·규모·일관성에 적합한지 |
| **부작용 분류** | 1층/2층/3층 계층 분류가 정확한지 |
| **용량 정보** | 권장/상한 용량이 출처와 일치하는지 |
| **상호작용** | 상호작용 내용·심각도 분류가 정확한지 |
| **claim 연결** | 원료↔기능성 연결이 근거에 기반하는지 |
| **라벨 파싱** | 파싱된 성분표가 라벨 원문과 일치하는지 |

#### 판정
- **승인** → L3 review_task 자동 생성
- **반려** → `rejection_reason` 기록, 수정 요청 (L1부터 재시작)
- **조건부 승인** → 코멘트 첨부, L3로 전달 (L3에서 최종 판단)

---

### L3: 규제 검수 (RA 자문)

**담당**: 규제(RA) 전문가
**SLA**: 5~7일
**목적**: 규제 준수·표현 적법성 확인

#### 검토 항목

| 항목 | 검토 내용 |
|------|-----------|
| **허용 기능성 문구** | 식약처 인정 문구를 정확히 사용하는지 |
| **비허용 표현** | 질병 치료/예방 암시 표현이 없는지 |
| **과대광고 소지** | 근거 등급 대비 과장된 표현이 없는지 |
| **면책 문구** | 적절한 면책 표시가 포함되었는지 |
| **국가별 분리** | KR/US 규제 정보가 올바르게 분리되었는지 |
| **특수 집단 주의** | 임산부/수유부/소아/고령자 주의사항 누락 없는지 |

#### 판정
- **승인** → `is_published = TRUE`, 서비스 노출
- **반려** → `rejection_reason` 기록, 수정 요청 (L2부터 재시작)
- **조건부 승인** → 표현 수정 후 즉시 게시 가능 (재검수 불필요)

---

## 3. 에스컬레이션 규칙

특정 변경은 반드시 해당 레벨 검수를 거쳐야 한다.

| 변경 유형 | 필수 검수 |
|-----------|-----------|
| 신규 원료 추가 | L1 → L2 → L3 (전체) |
| 신규 제품 추가 | L1 → L2 → L3 (전체) |
| 근거 등급 변경 | L2 필수 |
| 허용 기능성 문구 변경 | L3 필수 |
| 부작용/경고 신규 추가 | L2 + L3 모두 필수 |
| 라벨 변경 감지 | L1 → L2 → L3 (전체 재검수) |
| 단순 오타 수정 | L1만 (자동 통과 가능) |
| 출처 URL 변경 | L1만 |

## 4. review_tasks DB 연동

### task 생성 규칙

```
1. 데이터 입력/수집 시 L1 task 자동 생성
   - review_level = 'L1'
   - parent_task_id = NULL
   - status = 'pending'

2. L1 통과 시 L2 task 자동 생성
   - review_level = 'L2'
   - parent_task_id = L1 task id
   - assigned_role = 'scientific_reviewer'

3. L2 승인 시 L3 task 자동 생성
   - review_level = 'L3'
   - parent_task_id = L2 task id
   - assigned_role = 'regulatory_reviewer'

4. L3 승인 시
   - 대상 엔티티 is_published = TRUE
   - revision_histories 기록
```

### SLA 기한 자동 설정

| 레벨 | due_at 계산 |
|------|-------------|
| L1 | created_at + 1 day |
| L2 | created_at + 5 days |
| L3 | created_at + 7 days |

### 알림 규칙
- SLA 50% 경과: 담당자에게 리마인더
- SLA 초과: PM에게 에스컬레이션 알림
- 적체 50건 초과: 대시보드 경고

## 5. 역할 정의

| 역할 | 접근 권한 | 검수 권한 |
|------|-----------|-----------|
| **QA** | raw 문서, 추출 결과, L1 태스크 | L1 수동 검토, 수정, 재실행 |
| **Scientific Reviewer** | 모든 데이터, L2 태스크 | L2 승인/반려, 근거 등급 수정 |
| **Regulatory Reviewer** | 모든 데이터, L3 태스크 | L3 승인/반려, 표현 수정 요청 |
| **Admin** | 전체 | 모든 레벨 검수, 역할 할당, 정책 변경 |
